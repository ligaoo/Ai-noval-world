# MVP V2 总览与拆分（V2.1 / V2.2 / V2.3）

> 目的：避免“一次升级太多导致难定位”，将 V2 拆成 3 个可验证的小版本，每一步只引入少量新变量，保证可控、可调试、可复现。

## 1. 总体边界（所有 V2.x 都必须遵守）
- 人物 Agent **只输出结构化动作 ActionCommand JSON**，禁止输出小说正文
- 环境引擎（规则引擎）是唯一事实生成者：动作合法性/成功与否/线索触发/状态变化都由规则判定
- 叙事层只做“事件日志改写”，不得新增线索/对象/地点/真相
- 一致性检查分两层：RuleCheck（可程序判定优先）+ LLMCheck（语义层），并允许最多自动修订一次
- 存储方式：**本地文件**，按 `outputs/sim_xxx/` 文件夹隔离管理

## 2. 三个小版本的分工

### V2.1：只接入 LLM Agent（仍单地点）
**核心验证点**：LLM 是否稳定输出 JSON、ActionValidator 是否能拦住乱编、fallback 是否能保证不中断。  
不引入：多地点、记忆、LLM 改写。

### V2.2：加入多地点 + move（仍不做复杂记忆/LLM 改写）
**核心验证点**：地图联通、位置隔离、跨地点线索、移动轨迹回放、跨地点对话禁止等。  
LLM Agent 可以继续使用（推荐沿用 V2.1 的稳定版）。

### V2.3：加入记忆 + LLM 小说改写 + 一致性修订（强化“像小说”）
**核心验证点**：记忆是否减少重复动作、章节改写是否只基于 plot events、实体白名单与一致性检查是否能拦新增事实，并稳定修订一次。

## 3. 交付节奏建议
- 先完成 V2.1（稳定性与可控性）
- 再完成 V2.2（世界状态机与空间扩展）
- 最后完成 V2.3（写作质量与一致性兜底）

> 你可以把每个小版本当作“一个可合并 PR 的里程碑”：每步都能跑通并生成可验证产物（events/chapter/report）。
