# AI Noval World V1.1 开发计划（章节生成约束与质量闭环）

## 1. 目标与范围

### 1.1 版本目标
V1.1 目标是把“第一章暴露问题”沉淀为**系统级规则与自动化修复链路**，而不是一次性手工改文。

### 1.2 本版本覆盖问题
1. 时间线矛盾（废弃两年/九年/十年混乱）
2. 线索密度过高（第一章像线索清单）
3. 主角情感动机薄弱（调查工具人）
4. NPC 对话信息泄露过多（像任务发放器）
5. 描写过度修辞化（比喻堆叠）
6. 第一章灵异钩子弱或不稳定

### 1.3 影响链路
`ChapterPlanner -> NarrativeWriter -> StoryQualityEvaluator -> RewriteOptimizer -> ConsistencyCheck -> Horror/Mystery Genre Pack`

---

## 2. 实施策略（V1.1）

### 2.1 实施原则
1. 优先加“结构化字段 + 规则检查”，再改 prompt。
2. 所有新规则必须可被评估器识别，并自动转成 rewrite_task。
3. 第一章策略独立（OpeningChapterPolicy），后续章节策略复用。

### 2.2 交付方式
1. 新增配置字段（world_bible/characters/style/quality policy）
2. 新增 checker/controller 服务
3. 扩展现有 dataclass 与枚举
4. 扩展评估问题类型与修稿任务映射
5. 在 orchestrator 流程中串联新模块

## 5. 模块开发计划（按链路顺序）

## 5.1 ChapterPlanner 阶段

### 5.1.1 OpeningChapterPolicy（新增）
文件：`app/services/opening_chapter_policy_service.py`

职责：
1. 检测 `chapter_no == 1`
2. 注入第一章约束（线索上限、必须元素、禁用元素、轻异常钩子）
3. 生成/补齐 `chapter_plan.clue_budget`、`required_character_beats`、`ending_hook`

建议接口：

```python
class OpeningChapterPolicyService:
    def apply(self, chapter_plan: dict, world_bible: dict, characters: dict) -> dict:
        ...
```

### 5.1.2 ChapterClueBudgetController（新增）
文件：`app/services/chapter_clue_budget_controller.py`

职责：
1. 读取 `chapter_plan.clue_budget`
2. 从候选线索中筛出 `selected_clues`
3. 其余放入 `reserved_clues`
4. 保证 `must_include_clues` 不被移除

建议接口：

```python
class ChapterClueBudgetController:
    def apply(self, chapter_plan: dict, available_clues: list[dict]) -> dict:
        ...
```

### 5.1.3 orchestrator 接入
文件：`app/services/novel_production_orchestrator.py`

在 `_produce_next_chapter()` 中新增顺序：
1. resolve chapter function
2. if chapter1: apply opening policy
3. apply clue budget
4. pass enriched plan to chapter_generator

---

## 5.2 NarrativeWriter 阶段

文件：`app/services/narrative_writer.py`

改动点：
1. `_build_plan()` 读取并写入：`clue_budget`、`required_character_beats`、`ending_hook`
2. `_write_draft()` 增加三段“硬约束”注入：
   1. `Required Character Beats`
   2. `NPC Dialogue Policy`
   3. `Ending Hook`
3. 第一章生成中，强制至少 1 个短情感 beat（限制字数）

说明：当前实现是模板改写模式，不是完整 LLM planner。V1.1 可先通过“plan 字段驱动模板输出 + 后置质量检查”完成闭环。

---

## 5.3 ConsistencyCheck 阶段

### 5.3.1 TimelineConsistencyChecker（新增）
文件：`app/services/timeline_consistency_checker.py`

职责：
1. 从 world_bible.timeline 读取标准时间
2. 抽取章节中的时间表达（正则 + 关键词）
3. 判断“时间表达 -> 语义槽位”是否冲突
4. 输出结构化违规项，供 RewriteOptimizer 生成 `fix_continuity`

建议输出：

```json
{
  "passed": false,
  "issues": [
    {
      "type": "timeline_conflict",
      "phrase": "废弃两年",
      "expected_slot": "official_closed_years_ago",
      "expected_value": 2,
      "severity": "high"
    }
  ],
  "rewrite_suggestion": "统一时间线：十年前事故，九年前实际荒废，两年前官方登记停用。"
}
```

### 5.3.2 接入 ConsistencyCheckService
文件：`app/services/consistency_check.py`

新增顺序：
1. 现有 rule_check
2. timeline_check
3. 汇总 violations

新增 violation type：
1. `timeline_conflict`
2. `timeline_ambiguous_statement`

---

## 5.4 StoryQualityEvaluator 阶段

文件：
1. `app/quality/story_quality_evaluator_service.py`
2. `app/quality/quality_problem_classifier.py`
3. `app/quality/rewrite_suggestion_generator.py`

新增问题类型：
1. `clue_overload`
2. `missing_required_character_beat`
3. `thin_emotional_motivation`
4. `metaphor_overload`
5. `decorative_description`
6. `weak_horror_hook`

核心规则：
1. `discovered_clue_count > clue_budget.max_total` -> `clue_overload`
2. `required beat not found` -> `missing_required_character_beat`
3. 比喻词密度超阈值 -> `metaphor_overload`
4. 第一章 ending hook 缺失或不属于 allowed subtle devices -> `weak_horror_hook`

---

## 5.5 RewriteOptimizer 阶段

文件：`app/services/rewrite_optimizer.py`

### 5.5.1 扩展问题映射
在 `_problem_to_rewrite_task()` 映射新增：
1. `timeline_conflict -> FIX_CONTINUITY`
2. `clue_overload -> REDUCE_CLUE_DENSITY`
3. `metaphor_overload -> REDUCE_OVERWRITING`
4. `missing_required_character_beat -> DEEPEN_CHARACTER`

### 5.5.2 扩展 REWRITE_TASK_PROMPTS
新增任务模板：
1. `FIX_CONTINUITY`
2. `REDUCE_CLUE_DENSITY`
3. `REDUCE_OVERWRITING`

约束建议：
1. `REDUCE_CLUE_DENSITY`: 不删 must_include，优先后移 reserved
2. `REDUCE_OVERWRITING`: 保留信息性环境细节，删除装饰性重复比喻
3. `FIX_CONTINUITY`: 不改事件，只改表述与归因

## 8. 验收标准（Definition of Done）

### 8.1 功能验收
1. 第一章计划输出有 `clue_budget`、`required_character_beats`、`ending_hook`。
2. 出现“废弃两年/九年/十年混用”时，ConsistencyCheck 报 `timeline_conflict`。
3. 第一章线索数超预算时，QualityReport 报 `clue_overload`。
4. 缺少 `beat_last_call_memory` 时，QualityReport 报 `missing_required_character_beat`。
5. 比喻密度超阈值时，QualityReport 报 `metaphor_overload`。
6. RewritePlan 可自动生成 `fix_continuity/reduce_clue_density/reduce_overwriting`。

### 8.2 回归验收
1. 旧流程仍可跑通（无新增字段时默认值不报错）。
2. 未启用 LLM 时，规则检查与报告仍可输出。
3. Horror 非第一章不受 opening policy 强限制。

## 10. 风险与缓解

1. 风险：当前代码里存在编码异常文本（中文注释乱码），可能影响规则关键词匹配。
   1. 缓解：新增规则优先使用独立关键词表，不依赖乱码注释。
2. 风险：模板写作模式下情感 beat 注入效果不稳定。
   1. 缓解：先由 evaluator 严格兜底，rewrite 二次修正。
3. 风险：过度限制导致文本僵硬。
   1. 缓解：把阈值放在 quality_policy，可按项目调参。
