# V3.3 开发计划：ChapterContinuityService

> 版本主题：多章节连续性与未解决线索追踪  
> 目标：让章节之间保持连续，确保已发现事实、未解决问题、人物状态和下一章种子能够稳定继承。

---

## 1. 背景

V3.2 解决了剧情阶段和线索曝光问题，但多章生成仍可能出现：

```text
下一章忘记上一章发生了什么
已发现线索丢失
已解决问题又被当成悬念
未解决问题没人继续追
人物目标每章重置
章节之间像独立短篇
```

V3.3 要引入 ChapterContinuityService。

它的核心是：

> 每章结束后生成结构化摘要，下一章不依赖全文，而依赖结构化连续性上下文。

---

## 2. 版本目标

实现：

```text
ChapterSummary
OpenThreadTracker
ResolvedThreadTracker
NextChapterSeed
ContinuityContextBuilder
ChapterContinuityCheck
```

系统应支持：

```text
1. 每章结束生成 chapter_summary
2. 记录 new_facts / new_beliefs
3. 记录 open_threads / resolved_threads
4. 生成 next_chapter_seeds
5. 下一章 AgentContext 注入 continuity context
6. 下一章 chapter_plan 必须承接至少一个 seed
7. 防止已解决悬念反复出现
```

---

## 3. 不做什么

V3.3 不做完整长篇规划器。

暂不做：

```text
几十章全局大纲
自动章节分卷
复杂多主线编排
多 POV 章节调度
```

V3.3 只解决相邻章节连续性。

---

## 4. 核心概念

### 4.1 open_threads

尚未解决的问题。

例如：

```text
谁换了医院大门的锁？
看门人为什么隐瞒？
档案室为什么被封？
```

---

### 4.2 resolved_threads

已经解决的问题。

例如：

```text
铁锁是否最近更换过？是。
旧医院是否完全废弃？不是。
```

---

### 4.3 next_chapter_seeds

下一章可以继续推进的方向。

例如：

```text
追问看门人锁的来源
调查医院大厅前台
寻找旧维修单
```

## 6. 模块设计

## 6.1 ChapterContinuityService

### 职责

```text
1. 根据 chapter_plan / chapter_draft / EventLog 生成 ChapterSummary
2. 抽取 new_facts
3. 抽取 new_beliefs
4. 维护 open_threads
5. 维护 resolved_threads
6. 生成 next_chapter_seeds
7. 构建下一章 ContinuityContext
8. 检查下一章是否承接上一章
```

### 核心方法

```python
class ChapterContinuityService:
    def summarize_chapter(self, simulation_id: str, chapter_id: str) -> ChapterSummary:
        pass

    def extract_open_threads(self, events: list[EventLog], draft: str) -> list[Thread]:
        pass

    def resolve_threads(self, previous_threads: list[Thread], events: list[EventLog]) -> list[Thread]:
        pass

    def generate_next_chapter_seeds(self, summary: ChapterSummary) -> list[ChapterSeed]:
        pass

    def build_continuity_context(self, simulation_id: str, chapter_no: int) -> ContinuityContext:
        pass

    def check_chapter_continuity(self, chapter_plan: ChapterPlan, context: ContinuityContext) -> ContinuityReport:
        pass
```

## 7. Thread 状态管理

Thread 状态：

```text
open
in_progress
resolved
abandoned
blocked
```

Thread 优先级：

```text
1–10
```

推荐规则：

```text
主线相关 thread 优先级 8–10
人物关系 thread 优先级 5–7
氛围/支线 thread 优先级 3–5
```

---

## 8. Thread 解决规则

一个 thread 可以通过以下方式 resolved：

```text
某个 clue 被发现
某个 NPC 透露信息
某个 EventLog 明确回答问题
PlotArc stage 推进导致该问题完成
人工标记 resolved
```

示例：

```json
{
  "thread_id": "thread_001",
  "question": "谁换了医院大门的锁？",
  "resolved_by": "evt_0041",
  "answer": "看门人承认是一个陌生男人让他换的。",
  "status": "resolved"
}
```

---

## 10. 下一章 chapter_plan 约束

生成下一章 chapter_plan 时，必须满足：

```text
至少承接一个 open_thread
至少使用一个 next_chapter_seed
不能把 resolved_thread 重新当作主悬念
不能丢失上一章 new_facts
```

## 13. 验收用例

### 用例 1：生成 chapter_summary

条件：

```text
ch_001 完成
```

预期：

```text
生成 summary / new_facts / open_threads / next_chapter_seeds
```

---

### 用例 2：下一章继承 open_threads

条件：

```text
ch_001 有 open_thread：谁换了锁？
```

预期：

```text
ch_002 AgentContext 包含该 open_thread
ch_002 chapter_plan 至少承接一个 seed
```

---

### 用例 3：resolved_thread 不重复

条件：

```text
问题“铁锁是否最近更换过”已 resolved
```

预期：

```text
ch_002 不能重新把它当作主悬念
ContinuityReport 能识别
```

---

### 用例 4：known_facts 不丢失

条件：

```text
ch_001 发现 hf_001
```

预期：

```text
ch_002 AgentContext 中仍包含 hf_001
```

## 15. DoD

```text
1. 每章结束生成 chapter_summary
2. chapter_summary 包含 new_facts / new_beliefs / open_threads / resolved_threads / next_chapter_seeds
3. chapter_summaries.jsonl 可持续追加
4. 下一章 AgentContext 能注入 ContinuityContext
5. 下一章 chapter_plan 至少承接一个 next_chapter_seed
6. resolved_threads 不能被重复当作主悬念
7. 多章生成时 known_facts / beliefs 不丢失
8. 连续生成 3 章时主线不重置
9. ContinuityReport 能指出章节断裂问题
```
