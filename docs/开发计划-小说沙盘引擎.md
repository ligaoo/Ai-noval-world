# 开发计划：多 Agent 小说沙盘引擎（World Simulation → Event Log → Narrative）

> 目标：不是“直接让 AI 写小说”，而是**先模拟世界中角色真实行动与碰撞**，记录可追溯的事件日志（Event Log），再由叙事层在严格信息边界下将日志改写为小说文本，从而获得更强的一致性、可控性与可扩展性。

---

## 1. 产品定位与范围

### 1.1 产品形态
- AI 小说世界模拟器 / 剧情沙盘 / 角色驱动叙事引擎
- 核心闭环：**用户给设定 → Agent 跑一晚 → 生成事件日志 → 改写成小说章节 → 一致性检查**

### 1.2 目标（Goals）
- 以结构化世界设定驱动模拟，确保“世界规则可判定、事件可追溯”
- 人物 Agent 严格遵守知识边界（Fog of War），避免上帝视角
- 环境裁判（规则引擎 + 受限 LLM）裁定动作结果，生成统一事件日志
- 叙事层只做“改写与编排”，禁止新增核心事实/线索
- 形成可迭代的 MVP 路线：单地点 → 多地点+记忆 → 导演系统 → 群像多 Agent

### 1.3 非目标（Non-goals，第一版不做）
- 复杂战斗/经济系统/超大地图/几十个 Agent/长期连载
- 让 LLM 同时兼任人物、环境与作者（会导致乱编与逻辑崩溃）

---

## 2. 总体架构（7 层）

```text
1. 世界设定层 World Bible
2. 世界状态层 World State
3. 环境裁判层 Environment Engine
4. 人物 Agent 层 Character Agents
5. 事件调度层 Simulation Scheduler
6. 事件日志层 Event Log
7. 小说生成层 Narrative Writer
```

### 2.1 核心原则（必须强约束）
- Agent **不能编剧情**：只能基于自身信息做决策，输出结构化动作
- 环境引擎 **负责判定与后果**：动作是否合法/成功/触发线索/更新状态
- 小说层 **只负责写好看**：只能基于 Event Log 改写，不得新增真相/线索

---

## 3. 核心数据与结构化建模

### 3.1 World Bible（世界圣经）
- 字段建议：world_id、genre、tone、era、rules、themes
- `rules` 用于环境裁判判定（不是给小说看的）

### 3.2 地图与地点（Locations）
每个地点至少包含：
- public_description（公开描述）
- hidden_facts（隐藏信息 + 发现条件 discover_condition）
- objects（可交互对象）
- connected_to（连接地点）
- danger_level（危险等级）
- time_effects（时间变化）

### 3.3 物品（Objects）
- 可见性、发现条件、效果（effects）、世界观补充（lore）

### 3.4 世界事实分级（解决“信息不全”）
- A Truth（世界真相，仅环境/导演可见）
- B Visible Facts（当前可见）
- C Hidden Facts（可被发现）
- D Rumors（谣言/误解）

### 3.5 Fog of War（Agent 知识状态）
每个 Agent 持有独立 knowledge state：
- known_facts / suspicions / false_beliefs / unknowns
- 决策输入只包含：人物设定、当前可见信息、自己记忆、目标、附近人物、可用动作

---

## 4. Agent 设计（人物不是 prompt，而是对象）

### 4.1 Character Profile（人物画像）
建议字段：
- personality（traits + Big Five）
- goals（long_term / short_term）
- fears / secrets
- skills（observation/social/courage/logic…）

### 4.2 记忆系统（5 类）
- episodic_memory：具体经历事件
- semantic_memory：总结出的事实
- emotional_memory：情绪痕迹
- relationship_memory：对他人的看法
- plan_memory：计划与未完成任务

写入来源：Event Log（每个事件可生成 0..n 条记忆）

### 4.3 行动循环（每 Tick）
```text
Observe → Retrieve Memory → Think → Plan → Act
→ Environment Judge → Update Memory
```

### 4.4 动作输出（必须结构化）
人物 Agent 输出 JSON（ActionCommand），禁止输出小说正文。

---

## 5. 环境裁判（Environment Engine）

### 5.1 职责
- 动作合法性判断（地点/目标/可见性/工具/状态）
- 成功判定（规则检验、技能检定）
- 触发隐藏信息（discover_condition）
- 更新世界状态（时间、地点、物品、角色状态）
- 生成 Event Log（单一事实来源）

### 5.2 两层裁判
1) **规则引擎（确定性）**：位置、门锁、钥匙、状态、已发现线索等  
2) **受限 LLM 裁判（模糊语义）**：将自然语言意图映射到标准动作；生成合理感官描述  
> 约束：LLM 不能创造未提供的核心事实；只能在现有世界状态上做解释与描述。

---

## 6. Action Schema（动作白名单）

### 6.1 MVP1 动作集（建议）
- move / observe / inspect / search / talk / ask / wait

### 6.2 统一动作格式（示例）
```json
{
  "action_type": "search",
  "actor": "char_linzho",
  "target": "front_desk",
  "location": "hospital_lobby",
  "parameters": {
    "intensity": "careful",
    "tool": "phone_light"
  }
}
```

### 6.3 校验策略（必须）
- JSON Schema 校验：字段完整、类型正确、枚举合法
- Action 白名单：不在列表内直接拒绝执行
- Knowledge Boundary 检查：若含有“角色不可能知道的信息”，拒绝或要求重决策

---

## 7. Event Log（事件日志：唯一事实来源）

### 7.1 事件结构（关键字段）
- event_id / time / location / actors / event_type
- action（请求）/ result（结果）
- visible_to（可见者）
- hidden_effects（隐藏影响）
- emotional_impact（角色情绪）
- plot_value（mystery/conflict/progress…）

### 7.2 分级与入稿策略
- 低价值事件（无意义移动/重复询问/无戏剧价值失败）不进入小说
- 每章核心事件建议：5–12 条

---

## 8. Narrative Writer（小说生成层：日志改写，不写剧情）

### 8.1 四步流水线
1) 事件筛选（过滤低价值事件）
2) 章节编排（Chapter Plan）
3) POV 选择与信息禁泄露（forbidden_information）
4) 小说化写作（只做描写、节奏、心理、氛围增强）

### 8.2 一致性审查（Consistency Check）
对生成正文进行自动审查：
- 是否新增线索/真相
- 是否泄露 POV 不知道的信息
- 是否改变事件顺序与结果
- 是否违背世界规则/人物动机

---

## 9. Director Agent（导演系统：控节奏，不控人物）

### 9.1 触发条件（张力指标）
每轮模拟后计算（可先用规则/启发式）：
- conflict_score / mystery_score / progress_score / relationship_score / danger_score / novelty_score

当连续 N 轮 progress/conflict 过低时，允许投放事件：
- 电话、停电、NPC 出现、时间限制、环境异变（必须符合世界规则）

### 9.2 禁止项
- 直接告诉真相
- 强行让反派自爆
- 让角色违背性格行事

---

## 10. 后端服务拆分（建议）

> 可用 Java/Spring Boot 或 Python 实现；先按服务职责拆清楚，后续可合并为模块以降低 MVP 成本。

- NovelWorldService：世界设定 CRUD、版本管理
- WorldStateService：世界状态加载/更新/快照
- CharacterAgentService：组装 Agent 上下文、调用 LLM、解析动作
- EnvironmentEngineService：动作裁判、状态变更、事件生成
- SimulationSchedulerService：Tick 调度、角色激活策略、时间推进
- MemoryService：写入/检索/反思/遗忘/去重/矛盾处理
- DirectorService：张力评估与干预
- EventLogService：事件存储、查询、索引
- NarrativeWriterService：事件筛选、章节编排、正文生成
- ConsistencyCheckService：事实一致性与边界审查

---

## 11. 存储设计（可先单体，后期拆分）

### 11.1 MySQL（结构化核心表）
- novel_world
- novel_character（profile_json / state_json）
- novel_location（location_json）
- novel_event_log（event_json + 索引：simulation_id、world_id+time）

### 11.2 MongoDB（复杂快照）
- simulation_state_snapshot：按 tick/时间保存世界与角色状态
- 便于回放、调试、断点续跑

### 11.3 向量库（记忆检索）
- 选型：pgvector / Qdrant / Milvus（优先易集成）
- 记忆打分（示例）：
  - score = similarity * 0.5 + importance * 0.25 + recency * 0.15 + emotion * 0.1

---

## 12. 对外接口与内部契约（建议先定契约）

### 12.1 核心对象契约
- ActionCommand（人物输出）
- ActionResult（环境裁判输出）
- EventLog（事实记录）
- ChapterPlan / ChapterDraft（章节产物）

### 12.2 API（示例）
- POST /worlds：创建世界（World Bible + 地图 + 人物）
- POST /simulations：创建模拟（选择世界、初始时间、角色集合）
- POST /simulations/{id}/tick：跑一轮 tick（或跑 N 轮）
- GET  /simulations/{id}/events：查询事件日志
- POST /simulations/{id}/chapters/{no}:generate：生成章节（基于事件窗口）
- POST /chapters/{id}:check：一致性审查

---

## 13. MVP 路线（分 4 阶段交付）

### MVP 1：单地点、双 Agent、短篇悬疑（最小闭环）
- 地图：旧医院（可只含大门/大厅，也可合并为 1）
- Agent：主角 + 看门人（或 2 个主 Agent）
- 线索：3 个隐藏事实
- 输出：1 个章节（或 2–3k 字试写）
- 动作集：observe/inspect/search/talk/ask/move/wait
- 验证点：
  - 人物能在信息不全下行动
  - 环境能按规则释放线索
  - Event Log 可改写成可读小说

### MVP 2：多地点 + 记忆检索
- 地点：大门/大厅/档案室/四楼走廊/病房（5 个节点）
- 加入：物品、时间变化、线索依赖、向量检索记忆
- 增加：反思（reflection）与 plan_memory

### MVP 3：Director（解决无聊/卡死/节奏拖）
- 实装张力指标与干预白名单
- 关键线索至少 3 条发现路径（避免卡死）

### MVP 4：多 Agent 群像 + 多 POV
- 主角/记者/反派/警察/灵异存在
- 并行行动、信息差、关系变化
- 章节多 POV 组织与禁泄露

## 14. 关键算法与规则（第一版用“可解释的简单规则”）

### 14.1 技能检定
- d100 或 0..100 对比 difficulty
- 可加入状态修正（恐惧/受伤/光照不足等）

### 14.2 线索设计原则（防卡死）
- 每个关键线索 ≥ 3 条入口（搜索/对话/物品/事件投放/失败后的替代发现）

### 14.3 事件窗口与章节边界
可选策略：
- 按“模拟时间段”（如 20:00–22:00）生成章节
- 按“主线推进阈值”（progress_score 达到阈值）切章

---

## 15. Prompt 模板（建议落库版本化）

### 15.1 人物 Agent Prompt（要点）
约束：
- 你是人物，不是作者
- 不得编造未知信息
- 只输出 JSON 动作

输入拼装：
- 人物信息（profile）
- 当前状态（state）
- 当前地点可见信息（visible）
- 已知事实 + 检索记忆（retrieved）
- 可用动作白名单

### 15.2 环境裁判 Prompt（要点）
约束：
- 你是裁判，不是小说作者
- 只能基于 world_rules + 当前状态 + hidden_facts 判定
- 可生成感官描述，但不可改动核心事实

### 15.3 反思 Prompt（要点）
输出 JSON：
- 新事实、关系变化、目标变化、可能误解、下一步倾向

## 16. 工程实现计划（建议 4–6 周可跑通 MVP1/MVP2）

> 时间仅为建议，可按人力调整；每阶段都应“可跑可回放可出稿”。

### 第 1 周：数据结构与最小执行链路（MVP1 基础）
- 定义并固化 JSON Schema：WorldBible/Location/Object/Character/Action/EventLog
- 实现规则引擎雏形：合法性判断 + 技能检定
- 实现 EventLog 落库与查询（先用单库也可）

### 第 2 周：人物 Agent 决策闭环（MVP1 完成）
- CharacterAgentService：上下文构建、LLM 调用、动作解析与校验
- SimulationScheduler：Tick 驱动、时间推进、角色激活
- 跑通：2 Agent + 单地点 + 3 线索 + 事件日志输出

### 第 3 周：叙事层（章节改写 + 一致性审查 v1）
- 事件筛选与章节编排（ChapterPlan）
- POV 信息边界注入（forbidden_information）
- ConsistencyCheckService v1（规则 + LLM 审查可选）

### 第 4–5 周：MVP2（多地点 + 记忆）
- 加入地图连通与多地点状态
- MemoryService：记忆写入、向量检索、重要性/新近度
- 反思生成与 plan_memory

### 第 6 周：MVP3（Director）可选
- 张力指标与干预机制
- 线索多入口完善

## 17. 测试与验收（Definition of Done）

### 17.1 功能验收（MVP1）
- [ ] 角色输出严格为 ActionCommand JSON（schema 通过）
- [ ] 环境裁判不产生超出世界状态的“新真相”
- [ ] 每个动作必生成可追溯 EventLog（含 visible_to）
- [ ] 章节生成不泄露 POV 不知道的信息
- [ ] 一致性审查能识别“新增线索/泄露信息/违背规则”

### 17.2 可观测性与调试
- [ ] 支持按 simulation_id 回放事件序列
- [ ] 支持查看每次决策输入（脱敏）与输出
- [ ] 支持定位“动作被拒绝”的原因（可解释）

---

## 18. 风险清单与应对

1) **LLM 乱编**  
→ 五道防线：Schema 校验 / 动作白名单 / Knowledge Boundary / 世界规则一致性 / 小说生成后审查

2) **模拟失控或无聊**  
→ Chapter goal + 张力指标 + Director 干预白名单

3) **线索卡死**  
→ 关键线索 ≥ 3 入口；失败动作也能产出戏剧价值或替代线索

4) **成本过高**  
→ NPC 分层（主 Agent / 半 Agent / 背景 NPC）；减少每 tick 调用次数；缓存与批处理

---

## 19. 交付物清单（每阶段产出）

- 世界设定样例（World Bible + 地图 + 角色）
- 可运行模拟（simulation runner）
- 事件日志（JSON + 数据库）
- 章节大纲（ChapterPlan）
- 小说章节正文（ChapterDraft）
- 一致性审查报告（违规点 + 建议修复）

## 20. 下一步建议（落地第一版内容）

第一版推荐主题：**单元剧灵异短篇模拟器**
- 题材：灵异悬疑
- 地图：5 个地点
- 人物：3 个
- 隐藏线索：10 个
- 核心真相：1 个
- 模拟时间：1 晚
- 输出：1 万字短篇（可先从 2–3k 字章节试跑）
