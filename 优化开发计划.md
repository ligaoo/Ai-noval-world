# 优化开发计划

## 目标

提升生成系统的可靠性，让章节输出严格建立在模拟状态、事件日志、角色关系、线索暴露和可见性边界之上。

核心目标：

- Writer 只文学化上游结构化结果，不自行创造剧情级事实、线索、规则或关系变化。
- 章节完成度由可验证状态变化决定，而不是由文本观感决定。
- Agent 互动必须产生可持续影响，包括关系、信念、立场、目标或记忆变化。
- 所有输出 artifact 保持一致，包括 `state.json`、`metrics.json`、`plot_arc_state.json`、`chapter_plan.json`、`quality_report.json` 和 `run_status.json`。
- QA、可见性、状态闭合和草稿忠实度检查能阻止不可靠结果被标记为成功。

---

## 当前主要问题

### 1. 章节目标完成度过于宽松

现状：

- `state.json` 可将章节目标标记为 100% 完成。
- 但实际线索发现、关系变化、目标推进可能不足。
- Writer 会用文学化内容补足缺失推进。

风险：

- 系统接受“看起来完成，但状态未完成”的章节。
- 后续章节缺少可靠状态基础。

### 2. Writer 引入未授权实体或事实

现状：

- 草稿中可能出现未在 world/state/event 中登记的新道具、新地点、新规则、新线索。
- consistency report 即使发现 `new_entity`，也可能低危通过。

风险：

- Writer 绕过模拟层推进剧情。
- 状态机与正文内容脱节。

### 3. Agent 行为重复且低信息密度高

现状：

- 事件中存在大量有效但无推进的重复对话或重复动作。
- invalid action rate 为 0，但 repeat action count 很高。

风险：

- Writer 收到大量低价值事件。
- 章节靠文笔补强，而不是靠状态推进。

### 4. 关系和私人动机系统未充分生效

现状：

- 章节中出现人际紧张，但 `relationship_update_count` 可能为 0。
- `private_tendency_trigger_count` 可能长期为 0。

风险：

- 人物冲突只存在于正文，不存在于状态。
- 后续章节无法继承真实关系变化。

### 5. POV / hidden actor 可见性边界不稳定

现状：

- 隐藏角色发现或知道的事实可能直接进入 POV 正文。
- 缺少明确的信息传递事件。

风险：

- 叙事泄露隐藏事实。
- 破坏悬疑、公平性和角色知识边界。

### 6. artifact 状态不同步

现状：

- `state.json`、`metrics.json`、`plot_arc_state.json` 可能对已发现线索给出不同结果。
- `llm_traces.jsonl` 与 `llm_summary.json` 可能不一致。

风险：

- 调试指标不可信。
- plot arc 无法稳定跨章节推进。

### 7. QA 失败未阻止 run completed

现状：

- quality evaluation 失败时，`run_status.json` 仍可能显示 `completed`。

风险：

- 验证失败被隐藏。
- 用户误以为结果可用。

### 8. planning 元数据可能乱码

现状：

- `chapter_plan.json` 中可能出现 mojibake。

风险：

- Writer 控制信号损坏。
- 调试报告不可靠。

---

## 总体实施策略

分阶段推进：

1. **P0：阻止错误结果被接受**
   - 增加验证和阻断机制。
   - 修复 artifact 不一致、QA 失败未上报、Writer 未授权创造等问题。

2. **P1：提升 Agent 事件有效性**
   - 降低重复低价值事件。
   - 强制关键 interaction 闭合到状态变化。
   - 优化 memory 写入质量。

3. **P2：增强 Writer 忠实度与长期连续性**
   - 加强 draft faithfulness scoring。
   - 加强 plot arc 与跨章节状态同步。
   - 增加更细粒度的授权事实投影。

---

# P0：可靠性与阻断机制

## P0.1 章节目标完成 Checklist

### 目标

章节完成度必须由明确状态条件决定，而不是仅靠 `progress_delta` 累计。

### 新增文件

- `app/services/chapter_goal_completion_checker.py`

### 修改文件

- `app/services/world_state_updater.py`
- `app/runner/simulation_runner.py`
- `app/models/state.py`

### 实现内容

新增 `ChapterGoalCompletionChecklist`：

```python
class ChapterGoalCompletionChecklist(BaseModel):
    required_clue_ids: List[str] = []
    required_visible_clue_ids: List[str] = []
    min_key_discussions: int = 0
    min_relationship_updates: int = 0
    min_belief_or_stance_updates: int = 0
    min_open_threads: int = 0
    required_location_ids: List[str] = []
```

新增 checker：

```python
class ChapterGoalCompletionChecker:
    def evaluate(state, events, checklist) -> ChapterGoalStatus
```

完成条件示例：

- 必须发现指定线索。
- 必须至少触发一次 key discussion。
- 必须至少产生一个关系、信念、立场或目标变化。
- 必须产生至少一个 unresolved thread 或 ending hook。

### 验收标准

- 如果线索发现不足，章节不能被标记为 100%。
- 如果没有关键讨论或状态变化，章节不能 completed。
- `metrics.json` 中记录 checklist 每项是否满足。

## P0.2 Writer 授权事实边界

### 目标

Writer 不得生成未授权的新剧情级事实、实体、规则、线索、地点、工具。

### 新增文件

- `app/services/writer_authorization_builder.py`
- `app/services/draft_faithfulness_checker.py`

### 修改文件

- `app/services/narrative_service.py`
- `app/services/consistency_service.py`

### 实现内容

在 `writer_structured_context` 中增加：

```json
{
  "authorized_entities": {
    "characters": [],
    "locations": [],
    "objects": [],
    "facts": [],
    "clues": [],
    "rules": []
  },
  "atmosphere_allowed": true,
  "forbidden_fact_ids": [],
  "pov_known_fact_ids": [],
  "pov_visible_event_ids": []
}
```

Writer prompt 中明确：

- 可添加感官描写、心理活动、节奏停顿。
- 不可新增剧情级道具。
- 不可新增规则。
- 不可新增可验证线索。
- 不可新增地点路线。
- 不可让未暴露事实变成正文真相。

### DraftFaithfulnessChecker 检查维度

- 新实体
- 新地点
- 新线索
- 新规则
- 新道具
- POV 泄露
- 关系状态偏差
- 时间顺序偏差

### 验收标准

- 新钥匙、新笔记、新规则等剧情级内容必须被判为 medium/high violation。
- high violation 会触发 revision 或标记 validation failed。
- `chapter_plan.json` 中能看到 authorized facts/entities。

## P0.3 QA 失败上报到 run_status

### 目标

质量评估失败不能被隐藏。

### 修改文件

- `app/runner/simulation_runner.py`
- `app/services/quality_evaluator.py` 或现有质量评估服务
- `app/models/run_status.py` 如存在

### 实现内容

将状态拆分为：

```json
{
  "generation_status": "completed",
  "validation_status": "failed",
  "status": "completed_with_validation_errors",
  "validation_errors": []
}
```

当 quality report 失败时：

- `run_status.status` 不再是纯 `completed`。
- `last_error` 或 `validation_errors` 写入失败原因。
- tuning report 必须引用 QA 失败。

### 验收标准

- quality report 出现 failed 时，run_status 必须反映。
- 不允许 `quality failed` 且 `run_status.last_error = null`。

# P1：Agent 状态闭合与事件质量

## P1.1 StateClosureValidator

### 目标

关键事件必须闭合到状态变化，不能只产生空转对话。

### 新增文件

- `app/services/state_closure_validator.py`

### 检查规则

对每个 key event：

必须至少满足一项：

- relationship update
- belief update
- stance update
- goal advanced / blocked
- fact exposure changed
- memory written
- group decision created

伪代码：

```python
if event.is_key_event:
    assert any([
        relationship_updates,
        belief_updates,
        stance_changes,
        goal_results,
        exposure_update,
        memory_write,
        group_decision,
    ])
```

### 验收标准

- key discussion 不能只有 generic dialogue。
- closure rate 写入 metrics。
- closure failed 的事件进入 tuning report。

---

## P1.2 低价值事件与重复行为抑制

### 目标

减少重复合法但无推进的行为。

### 新增文件

- `app/services/event_value_scorer.py`
- `app/services/agent_loop_breaker.py`

### Event Value Score

```python
event_value = (
    clue_delta * 4
    + relationship_delta * 3
    + belief_delta * 2
    + stance_delta * 2
    + goal_delta * 2
    + location_delta * 1
    + group_decision_delta * 2
    - repeat_signature_penalty
)
```

低价值事件定义：

```python
event_value <= 0
```

### Loop Breaker 策略

当同一 agent 连续重复：

- 同一 action_type
- 同一 target
- 同一 topic
- 无状态变化

超过 N 次后，强制：

- 换地点
- 换目标对象
- 换询问对象
- 升级为 challenge
- 触发 director intervention

### 验收标准

- repeat_action_count 高时自动介入。
- tuning report 显示 low-value event ratio。
- memories 不再大量记录重复无效对话。

## P1.3 Memory 写入质量过滤

### 目标

记忆只记录有长期价值的事件，而不是重复废话。

### 修改文件

- `app/services/memory_service.py`

### 实现内容

新增 memory quality gate：

```python
should_write_memory(event):
    return any([
        event.discovered_facts,
        event.relationship_delta,
        event.fact_exposure_delta,
        event.group_decision,
        event.goal_delta,
        event.is_unresolved_thread,
    ])
```

重复记忆合并：

- 相同 agent
- 相同 location
- 相似 content
- 相同 tick window

合并为：

```text
重复争论：围绕 X 的怀疑持续存在，但未得到验证。
```

### 验收标准

- memories.jsonl 不再充满重复 generic dialogue。
- 每条 memory 可追溯到 clue、relationship、belief、goal 或 open thread。

---

## P1.4 关系与私人倾向强制生效

### 目标

正文中的人际冲突必须有状态支撑。

### 修改文件

- `app/services/relationship_impact_evaluator.py`
- `app/services/private_tendency_trigger_check.py`
- `app/services/world_state_updater.py`

### 规则

以下行为必须产生关系或立场变化：

- challenge
- accuse
- withhold
- deflect
- deny
- interrupt
- block_disclosure
- secret_protection
- personal_grudge
- loyalty_override

示例映射：

```text
challenge -> hostility_rise 或 suspicion_rise
withhold -> suspicion_rise
support -> affinity_gain
protect -> affinity_gain + trust_gain
accuse -> suspicion_rise + hostility_rise
secret_protection -> observer suspicion_rise
```

### 验收标准

- interaction_count > 0 且 relationship_update_count 长期为 0 时触发 warning。
- private tendency trigger rate 写入 metrics。
- state.relationships 中 evidence 不为空。

# 推荐实施顺序

## 第一批：P0 阻断错误结果

1. `ArtifactConsistencyValidator`
2. `EncodingHealthChecker`
3. QA failure 写入 `run_status`
4. `WriterAuthorizationBuilder`
5. `DraftFaithfulnessChecker` 初版
6. 章节目标 checklist

完成后目标：

- 错误结果不会被误标记为 completed。
- Writer 新增剧情级内容会被拦截。
- artifact 不一致会被显式报告。

## 第二批：P1 提升模拟有效性

1. `StateClosureValidator`
2. `EventValueScorer`
3. `AgentLoopBreaker`
4. Memory 写入过滤
5. 关系/私人倾向强制映射

完成后目标：

- Agent 不再大量空转。
- interaction 产生持久状态变化。
- memory 更干净，后续 tick 更容易使用。

## 第三批：P2 长期稳定性

1. Draft faithfulness 完整评分
2. Plot arc/state 同步
3. Tuning report 升级
4. 跨章节状态一致性检查

完成后目标：

- 多章节运行更稳定。
- 调试报告能准确指出系统问题。
- Writer 输出与模拟状态保持长期一致。

构造：

- state discovered 1 clue。
- plot_arc_state discovered 0 clue。

断言：

- artifact consistency failed。

# 成功标准

完成优化后，一次章节生成应满足：

```text
1. 章节目标完成度可解释、可验证。
2. 线索发现、讨论、关系变化形成闭环。
3. Writer 没有创造未授权剧情事实。
4. POV 不泄露隐藏知识。
5. relationships / beliefs / stance / goals 至少有一种随关键事件变化。
6. artifact 状态一致。
7. QA 失败会影响 run_status。
8. tuning report 能指出真实系统问题。
9. memories 记录高价值因果事件，而不是重复废话。
10. chapter_plan 无乱码。
```

最终系统应达到：

> 章节不是由 Writer 编出来的，而是由状态机、Agent 行为、线索暴露和关系变化共同推出，再由 Writer 文学化呈现。
